# 并发性能

测试环境：Windows 11 + .NET 10 + SQLite + Redis (Docker)，使用 `curl` 并行请求模拟。

## 读操作（登录 — BCrypt + JWT + Redis）

| 并发数 | 总耗时 | 成功率 | 平均响应 |
|--------|--------|--------|----------|
| 10 | 305ms | 100% | 30ms |
| 50 | 716ms | 100% | 14ms |
| 100 | 1,326ms | 100% | 13ms |
| 200 | 2,508ms | 100% | 12ms |
| 500 | 5,535ms | 100% | 11ms |
| 2,000 | 24,105ms | 100% | 12ms |

## 瓶颈分析

```
Kestrel —— 默认无连接数限制，仅受系统资源约束
Redis  —— Token 验证缓存，有效降低 DB 压力
SQLite —— 仅支持单写者，EF Core 自动排队
JWT + BCrypt —— CPU 密集型，登录接口主要耗时来源

实测结论:
  • 读写混合 500~1000 并发无压力
  • 瓶颈在 SQLite 串行写入，写 >1000 延迟线性增长
  • 生产级高写入 → 推荐迁移 PostgreSQL / SQL Server
  • 更高读并发 → 加入 Response Caching 中间件
```
